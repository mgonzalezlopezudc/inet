### 1.6 PHY Layer — Error Model

The NIST and Yans error models have been extended with HE-specific branches:

- **Per-MCS modulation mapping** correctly maps MCS 0–11 to BPSK/QPSK/QAM-16/64/256/1024 in [getHeDataSuccessRate()](file:///home/user/omnetpp_ws/inet/src/inet/physicallayer/wireless/ieee80211/packetlevel/errormodel/Ieee80211NistErrorModel.cc).
- **LDPC SNR boost**: A flat **1.5 dB** multiplicative gain (`std::pow(10.0, 1.5 / 10.0)`) is applied to LDPC-coded data.

> [!WARNING]
> The 1.5 dB LDPC coding gain is a **reasonable simulation approximation** used in ns-3, but is not directly mandated by the standard. It varies by MCS and code rate in practice. This should be documented as a modeling choice.

> The HE-SIG-B RU allocation field is modeled via `Ieee80211HeSigCodec` encode/decode using catalog indices, which is a **simplification** of the standard's 8-bit B0–B7 RU allocation subfield encoding. The result is functionally equivalent for the modeled RU sizes but not bit-exact with the standard's Common field encoding.

### 2.2 Strengths

- **DL OFDMA** provides 7 configurations covering different scheduler strategies, bandwidth scaling (20→80 MHz), MU-MIMO, and Multi-TID — this is the most comprehensive example.
- **HE Features** is well-designed as a feature matrix: it isolates individual HE PHY features (LDPC, PE, puncturing) and then combines them.
- **BSS Coloring** provides a clear A/B comparison (disabled vs. enabled) with two overlapping BSSes.
- All `.ini` files follow the workspace rules: `[General]` at top, `description` on configs, units on all physical quantities, comments explain non-obvious choices.

### 2.3 Weaknesses

- **Short simulation times** (0.5–2 s). These are suitable for functional testing but insufficient for meaningful throughput/latency statistics.
- **No parameter studies.** No `repeat`, no `${variableName=...}` iterators. Cannot demonstrate capacity gains without running a campaign manually.
- **Missing `description` on some configs**: `ScheduledOnly`, `MixedUora`, `EqualRus` in `ul_ofdma/omnetpp.ini` lack descriptions.
- **No visualization configuration** beyond HE MU signal labels. No throughput visualizer, no data link visualizer, no statistics collection configuration for `opp_scavetool` post-processing.
- **Fixed MCS everywhere.** Rate adaptation is not demonstrated or even configurable. All examples use a single bitrate.
- **No fingerprint tests** attached to any example. The examples are not locked against regressions.

## 3. Code Quality

### 3.1 Comments & Documentation

**Rating: ★★★★☆ (Good)**

- Standard references are pervasive: nearly every struct, function, and constant carries an IEEE 802.11-2024 clause/table citation. Examples: every field in `Ieee80211HePhyCalculator.h`, every RU helper in `Ieee80211HeRu.h`.
- Class-level doxygen comments explain purpose, algorithms, and constraints (e.g., [HeHcf](file:///home/user/omnetpp_ws/inet/src/inet/linklayer/ieee80211/mac/coordinationfunction/HeHcf.h#L24-L38), [computeHePpduParameters](file:///home/user/omnetpp_ws/inet/src/inet/physicallayer/wireless/ieee80211/packetlevel/Ieee80211HePhyCalculator.h#L252-L259)).
- Inline comments explain why, not just what.

**Gaps:**
- Some anonymous helper functions in `HeHcf.cc` (e.g., `isMuEligibleDataHeader`, `hasEligibleExistingFrame`) lack any documentation.
- The MCS table static declarations in `Ieee80211HeMode.h` (400+ lines of near-identical declarations) have no per-entry comments and could benefit from code generation or a table-driven approach.

### 3.2 Programming Practices

**Rating: ★★★☆☆ (Adequate with concerns)**

**Positive:**
- Non-throwing validation API: `Ieee80211HePhyValidationResult` with `operator bool()` and descriptive `.error` strings is excellent.
- Liberal use of `ASSERT()` in the MAC layer.
- `EV_INFO` / `EV_DEBUG` logging at decision points throughout HeHcf.
- `DelayedInitializer` for the MCS table avoids static initialization order issues.
- Clean separation between scheduling interface (`IIeee80211HeDlScheduler`) and implementations.

**Concerns:**

- **Header-only calculation engine.** `Ieee80211HePhyCalculator.h` (547 lines), `Ieee80211HeRu.h` (379 lines), and `Ieee80211HeSigCodec.h` (106 lines) are all `inline` functions in header files. This increases compile times and couples all includers to the implementation.
- **God class tendency in HeHcf.** At 1461 lines, [HeHcf.cc](file:///home/user/omnetpp_ws/inet/src/inet/linklayer/ieee80211/mac/coordinationfunction/HeHcf.cc) handles DL scheduling context collection, UL trigger construction, sounding initiation, triggered UL response management, MIMO CSI management, and SU fallback. This is fragile and difficult to test in isolation.
- **Anonymous namespace helpers** at file scope (lines 40–196 of `HeHcf.cc`) are effectively non-testable private functions.
- **`#define DI DelayedInitializer`** at global scope in the header is a macro hygiene issue.
- **MCS table is manually enumerated** — 384+ static members typed out individually. This is maintenance-heavy and error-prone.
- **Magic numbers** remain in some places (e.g., `heSigBDataBitsPerSymbol = 26`, `21 * requestedUsers.size()` for user field bits).

### 3.3 Clarity & Readability

**Rating: ★★★★☆ (Good)**

- Naming conventions are consistent with INET (PascalCase classes, camelCase methods/variables).
- Enum values use descriptive names (`HE_GI_0_8_US`, `HE_CODING_LDPC`).
- The scheduler interface is clean and self-documenting via `CandidateInfo` and `ScheduleContext` structs.

### 3.4 Reusability

**Rating: ★★★★☆ (Good)**

- The scheduler is **fully pluggable**: `IIeee80211HeDlScheduler` and `IIeee80211HeUlScheduler` interfaces with NED-configurable implementations.
- Three DL schedulers (EqualSizedRUs, BacklogBased, HoLMinDelay) and two UL schedulers (EqualSizedRUs, BacklogBased) are provided.
- The PHY calculator is a pure function — no module dependencies.
- `Ieee80211HeCapabilities` struct + `negotiateHeCapabilities()` are reusable without MAC module coupling.

### 3.5 Debugging Ease

**Rating: ★★★★★ (Excellent)**

- Rich `EV_INFO` / `EV_DEBUG` logging throughout the MAC scheduling path.
- `ASSERT()` guards at critical decision points.
- Custom Qtenv visualization for HE MU signals (`displayHeMuSignalDetails`, `displayHeMuSignalPhyFields`).
- Non-throwing validation result with diagnostic error strings in PHY calculations.

## 6. SWOT Analysis

### Strengths

| # | Strength | Evidence |
|---|---|---|
| S1 | **Comprehensive HE feature coverage.** The implementation covers the core 802.11ax features: OFDMA (DL+UL), MU-MIMO, LDPC, BSS Coloring/Spatial Reuse, TWT, preamble puncturing, DCM, packet extension, and Multi-TID BA. | 80+ new source files, 5 example simulations, 39 unit tests |
| S2 | **Strong standard grounding.** Nearly every constant, timing value, and decision boundary cites an IEEE 802.11-2024 clause, table, or figure. | Pervasive `// IEEE 802.11-2024, Clause X.Y.Z` comments |
| S3 | **Pluggable scheduler architecture.** Clean interface (`IIeee80211HeDlScheduler`, `IIeee80211HeUlScheduler`) with 5 concrete implementations demonstrating different strategies. | NED-configurable `typename` in examples |
| S4 | **Correct PHY numerology.** DFT period, guard intervals, subcarrier counts, RU allocation tree, preamble field durations, and PPDU duration limit all match the standard. | Verified against 802.11-2024 Tables 27-61, 27-14, 27-64, 9-34 |
| S5 | **Excellent debugging infrastructure.** Rich EV logging, ASSERTs, non-throwing validation with diagnostic strings, custom Qtenv visualization. | `Ieee80211HePhyValidationResult`, `displayHeMuSignalDetails` |
| S6 | **Thorough test suite.** 39 unit tests (5500+ LOC) + 2 validation scripts cover PHY calculations, serialization, scheduling, Block Ack, spatial reuse, and MIMO. | `tests/unit/Ieee80211He*.test` |
| S7 | **Backward compatibility.** HE features are gated behind `opMode = "ax"`, `HeHcf`, and MIB flags. Existing 802.11a/g/n/ac simulations are unaffected. | `HeHcf` falls back to `Hcf::startFrameSequence()` |
| S8 | **On-wire serialization.** HE MU PHY header, Trigger Frame, and management elements have dedicated serializers with roundtrip unit tests. | `Ieee80211HeMuPhyHeaderSerializer`, `Ieee80211TriggerFrameSerializer_1.test` |

### Weaknesses

| # | Weakness | Impact |
|---|---|---|
| W1 | **No HE rate adaptation.** All examples use a fixed MCS. There is no HE-aware rate control module. | Users cannot simulate realistic link adaptation scenarios. |
| W2 | **HeHcf is a 1461-line god class.** It handles DL scheduling, UL triggering, sounding, MIMO CSI, TWT gating, and SU fallback in one module. | Difficult to test, maintain, and extend individual features. |
| W3 | **Header-only PHY calculator.** 1000+ lines of `inline` functions in headers increase compile time and couple all includers. | Slower builds, harder to benchmark in isolation. |
| W4 | **MCS table is manually enumerated** (384+ static members). | High risk of copy-paste errors; maintenance burden for any table change. |
| W5 | **SIFS 2.4 GHz is wrong (16 µs instead of 10 µs).** | Incorrect MAC timing for any 2.4 GHz HE simulation. |
| W6 | **No fingerprint tests on examples.** | Regressions can go undetected when modifying shared PHY/MAC code. |
| W7 | **Incomplete UORA.** OBO counter and random-access backoff are not modeled. | UL random access is not realistic. |
| W8 | **Short simulation times in examples** (0.5–2 s) and no parameter studies. | Cannot produce meaningful statistical results without user modification. |
| W9 | **HE-SIG-B encoding is a simplification** (catalog index, not bit-exact B0–B7 encoding). | Not interoperable with standard-compliant SIG-B decoders. |
| W10 | **1.5 dB LDPC gain is undocumented as a modeling assumption.** | Could mislead users into thinking it's a standard-mandated value. |

### Opportunities

| # | Opportunity | Potential |
|---|---|---|
| O1 | **HE rate adaptation module.** Implementing an MCS/NSS selection algorithm (e.g., Minstrel-HT adapted for HE) would unlock the most impactful research scenario. | High — rate adaptation is the #1 missing feature for realistic studies. |
| O2 | **Parameter studies / showcases.** Adding `repeat`, `${variableName=...}` iterators and result analysis scripts would demonstrate HE gains quantitatively. | Medium — would make the implementation immediately useful for research. |
| O3 | **6 GHz band (Wi-Fi 6E).** The infrastructure (HE mode, 160 MHz bands, 6 GHz capability element) is ready for 6 GHz channel definitions. | Medium — straightforward extension. |
| O4 | **802.11be (EHT / Wi-Fi 7) foundation.** The RU allocation tree, MU-MIMO CSI manager, and scheduler architecture are directly extensible to 320 MHz, multi-link, and enhanced OFDMA. | High — significant first-mover advantage. |
| O5 | **Bit-level PHY integration.** The packet-level model could be backed by a bit-level simulation for studies needing symbol-level accuracy. | Low — significant effort for marginal gain in most studies. |
| O6 | **Refactor HeHcf.** Extract UL coordinator, sounding, and DL scheduling into separate concerns. | Medium — would improve testability and maintainability. |

### Threats

| # | Threat | Risk Level |
|---|---|---|
| T1 | **Regression risk from HeHcf complexity.** Any change to the 1461-line HeHcf may break DL, UL, sounding, or TWT paths. | High — mitigated partially by unit tests but no integration/fingerprint tests. |
| T2 | **Compile-time impact.** Header-only PHY calculator + 700-line MCS table header increases incremental build times for any file including them. | Medium — affects developer iteration speed. |
| T3 | **Standard evolution.** IEEE 802.11be (Wi-Fi 7) introduces significant changes (320 MHz, multi-RU, multi-link) that may require architectural changes to the RU allocation and scheduler systems. | Medium — but the current architecture is extensible. |
| T4 | **Validation gap.** No comparison against ns-3, analytical models, or hardware measurements. The error model's LDPC gain and the scheduler's behavior are not validated against external references. | Medium — reduces confidence for research publications. |
| T5 | **Maintainability of manual MCS table.** Any standard correction or extension requires editing 400+ lines of near-identical declarations. | Low-Medium — error-prone but infrequent. |

## Summary

The 802.11ax implementation is a **substantial and largely correct** addition to INET, covering the key HE features (OFDMA, MU-MIMO, LDPC, BSS Coloring, TWT, preamble puncturing) with strong standard grounding and a clean scheduler architecture. The PHY numerology is correct. The test suite (39 unit tests) is comprehensive for a first implementation.

The main areas for improvement are:
1. **Add HE rate adaptation** — the single most impactful missing feature.
2. **Fix the 2.4 GHz SIFS** (10 µs, not 16 µs).
3. **Refactor HeHcf** to reduce class complexity.
4. **Add fingerprint tests** to lock example simulation outputs.
5. **Move PHY calculator out of headers** to reduce compile-time coupling.
6. **Document modeling assumptions** (LDPC gain, SIG-B simplification).

## 1. Executive Summary

The current 802.11ax implementation is a broad and mature HE feature set, not a narrow OFDMA prototype. It covers the central packet-level PHY and MAC mechanisms needed for Wi-Fi 6 studies:

- HE SU, HE ER SU, HE MU downlink, and HE trigger-based uplink PPDU formats.
- DL and UL OFDMA with pluggable schedulers.
- DL MU-MIMO modeling with CSI/sounding support.
- HE-SIG-B common-field RU allocation encoding using IEEE Table 27-27 style wire codes.
- LDPC, DCM, packet extension, preamble puncturing, 20/40/80/160 MHz channel support, and per-user PHY calculations.
- BSS coloring and OBSS/PD spatial reuse.
- Target Wake Time.
- Trigger frames, MU-BAR, BSRP, UORA random access, OBO/OCW state, and Multi-STA BlockAck.
- HE capability negotiation through management elements.
- HE-specific Minstrel-style rate adaptation.
- Buffer Status Report parsing/accounting for AP-side UL scheduling, including implicit BSR support.
- Example networks and walkthroughs for all major implemented HE features.

The biggest former functional gaps have been closed: HE rate adaptation exists, ER SU is modeled, and BSR is no longer merely a placeholder. The remaining issues are now more about model fidelity, maintainability, and regression coverage than missing first-order functionality.

The most important open risks are:

- No fingerprint or integration-regression tests for the example simulations.
- Several major algorithms remain high-level modeling approximations, especially LDPC coding gain, Minstrel thresholds, RU power/SNR estimation, UORA per-AC behavior, and scheduler heuristics.
- HE frame-sequence and DL MU assembly code is still large and intricate even after meaningful refactoring.
- Several large HE helpers remain header-heavy, especially the MCS table, RU utilities, and HE-SIG-B codec.
- Examples are useful and now documented, but most still run for short functional intervals and do not provide campaign-style parameter studies.

Overall verdict: **strong implementation, good standard grounding, good unit coverage, improving architecture, and credible simulation usability; still short of publication-grade validation against external models or hardware traces.**

## 2. Standard Compliance Review

### 2.1 PHY Timing and Numerology

| Topic | Current Implementation | Assessment |
|---|---|---|
| HE DFT period | `Ieee80211HeTimingRelatedParametersBase` defines the HE 12.8 us DFT period. | Correct. |
| Guard intervals | HE guard interval enum covers 0.8, 1.6, and 3.2 us variants. | Correct. |
| Symbol duration | Computed as DFT period plus selected GI. | Correct. |
| Slot time | HE mode set keeps the 9 us OFDM slot-time model for 5 GHz style operation. | Correct. |
| SIFS | `getSifsTime()` is band-aware: 10 us for 2.4 GHz and 16 us otherwise. | Correct. |
| PPDU duration limit | HE PPDU calculators enforce the 5.484 ms style maximum. | Correct. |
| HE ER SU preamble | `Ieee80211HePreambleMode::HE_PREAMBLE_ER_SU` adds duplicated HE-SIG-A duration; unit-tested in `Ieee80211HeErSuBsr_1.test`. | Correct for the modeled abstraction. |

The PHY timing path is now well annotated with standard clause references. The implementation distinguishes SU, ER SU, MU, and TB PPDU formats in the PHY calculator and headers.

### 2.2 RU Numerology and Allocation

| Topic | Current Implementation | Assessment |
|---|---|---|
| HE tone spacing | `HE_TONE_SPACING` uses 78.125 kHz. | Correct. |
| RU sizes | 26, 52, 106, 242, 484, 996, and 1992-tone RUs are represented. | Complete for 20/40/80/160 MHz HE. |
| Data and pilot subcarrier counts | `getHeRuDataSubcarrierCount()` and `getHeRuPilotSubcarrierCount()` encode the standard RU tables. | Correct. |

The RU helpers remain in a large header (`Ieee80211HeRu.h`, 372 lines), but functionally they are complete and unit-tested by `Ieee80211HeRu_1.test`.

### 2.3 MCS, NSS, DCM, and Coding

| Topic | Current Implementation | Assessment |
|---|---|---|
| MCS table | `Ieee80211HemcsTable` provides MCS 0-11 across bandwidth/NSS combinations. | Complete, but manually enumerated. |
| Code rates | MCS-to-code-rate mapping is implemented in HE helpers. | Correct. |
| Invalid MCS/NSS/RU combinations | `isHeValidMcsNssCombination()` rejects standard-invalid combinations and is used by the PHY calculator and rate control. | Correct. |
| DCM | DCM constraints are represented and rate control clamps DCM to robust MCS values. | Good. |
| BCC/LDPC | The PHY calculator enforces LDPC requirements for large RU/MCS combinations and BCC NSS limitations. | Correct. |
| Packet extension | PE duration is carried in HE common request/header state and examples cover it. | Good. |

The MCS table is the largest maintainability wart in the PHY mode layer. Its behavior is covered, but its source form is manually enumerated and difficult to audit for copy-paste mistakes.

### 2.4 HE PPDU Formats and PHY Headers

| PPDU Format | Current Support | Assessment |
|---|---|---|
| HE SU | Mode, preamble, data mode, PHY header, and normal MAC paths support HE SU. | Supported. |
| HE ER SU | Dedicated preamble format and PPDU format constant; rate control can request ER SU and restricts NSS/MCS accordingly. | Supported. |
| HE MU DL | Container packet with per-user RU payload headers, common HE MU request tag, and per-user PHY metadata. | Supported. |
| HE TB UL | Trigger-correlated receive collection, RU matching, Multi-STA BA generation, and per-MPDU receive-status parsing. | Supported. |

The PHY header serializer has a dedicated HE MU path with BSS color, PPDU format, GI/coding/PE fields, puncturing mask, and per-user RU information. `Ieee80211HeMuPhyHeaderSerializer_1.test`, `Ieee80211TriggerFrameSerializer_1.test`, and `Ieee80211OnWireBitCompliance_1.test` cover important serialization round trips.

### 2.5 HE-SIG-B Encoding

The HE-SIG-B common RU allocation path is no longer a simplified internal catalog. `Ieee80211HeSigCodec.h` implements Table 27-27 style encoding/decoding, including special cases such as 484/996-tone allocations and center 26-tone RU handling. The serializer uses this codec for HE MU PHY headers.

Assessment: **standard-compliant for the modeled common-field abstraction.**

Remaining concern: the codec is 629 lines and header-only. That is acceptable for correctness but suboptimal for compile-time and encapsulation.

### 2.6 Error Models

The NIST and Yans IEEE 802.11 error models include HE-specific branches:

- MCS 0-11 modulation mapping is represented.
- Per-user HE MU error evaluation is supported.
- LDPC-coded HE traffic receives a flat 1.5 dB style SNR gain.

Assessment: **good packet-level simulation model, but the LDPC gain is explicitly a modeling assumption rather than a standard rule.** It is useful for comparative simulation, but it should not be treated as a PHY-accurate LDPC decoder model.

### 2.7 Spatial Reuse and BSS Coloring

BSS color is propagated through MIB/operation state and HE PHY headers. `Ieee80211Receiver` implements OBSS/PD-style spatial reuse decisions, and the `bss_coloring` example provides an enabled/disabled comparison. There is a dedicated unit test, `Ieee80211HeSpatialReuse_1.test`.

Assessment: **good packet-level support.** It captures the decision-level behavior expected by users of INET, though it is not a hardware-calibrated CCA implementation.

### 2.8 TWT

TWT support includes an `Ieee80211TwtManager`, example agents/management modules, individual and broadcast examples, and gating in HE MAC scheduling paths. The current `twt` example runs for 100 s and includes a walkthrough.

Assessment: **implemented and demonstrable.** The TWT implementation is best understood as a functional MAC scheduling/sleep-state model, not a full power-consumption validation model.

### 2.9 UL OFDMA, BSR, BSRP, and UORA

UL OFDMA is now one of the stronger pieces of the implementation:

- `HeUlCoordinator` stores per-AID buffer-status state, freshness, retry flags, scheduled bytes, and service time.
- `HeUlDefaultTriggerPolicy` selects Basic or BSRP triggers based on associated STA and report freshness/backlog context.
- `HeUlMuTxOpFs` builds Basic/BSRP Trigger frames and collects HE TB responses in a strict trigger/RU/time window.
- Multi-STA BlockAck records are generated from received MPDUs.
- BSR data is added to HE STA uplink data and QoS-null style responses when needed.
- Implicit BSR from uplink SU data is supported.
- UORA uses an OBO counter and OCW update rule with reset on success and doubling on failure.

Assessment: **functionally strong, but with one important model simplification.** The UORA implementation explicitly uses a single OFDMA contention window/backoff state shared across access categories, while the standard maintains per-AC state. The code comments state this clearly. This is acceptable as a conservative simulation approximation, but it matters for QoS-sensitive UORA studies.

### 2.10 Rate Adaptation

The codebase now includes a HE-specific rate-control contract and a `HeMinstrelRateControl` implementation:

- `IIeee80211HeRateControl` exposes `selectHeMode()`, `reportHeTxResult()`, and `reportHeRxSnir()`.
- DL and UL schedulers can attach an optional `heRateControlModule`.
- The rate controller keeps per-peer, per-MCS/NSS EWMA success state.
- It periodically probes non-best rates via `lookaroundRatio`.
- It scores candidates by expected goodput.
- It can seed selection from SNIR using configurable thresholds.
- It honors standard constraints such as ER SU being single-stream and robust-MCS oriented, DCM clamping, RU-size limits, LDPC/BCC limits, and negotiated peer MCS caps.
- It emits `heRateSelectedMcs`, `heRateSelectedNss`, `heRateProbe`, and `heRateSuccessProbability` statistics.

Assessment: **the old fixed-MCS limitation is resolved.** The current controller is a useful Minstrel-style packet-level model, but it is still heuristic. The SNIR thresholds and EWMA behavior are not validated against hardware or ns-3 in this repository.

## 3. MAC Architecture Review

### 3.1 HE HCF

`HeHcf` remains the central integration point for HE MAC behavior. Its implementation has been split into:

| File | Lines | Responsibility |
|---|---:|---|
| `HeHcf.cc` | 350 | Initialization, timers, trigger access, high-level HCF integration. |
| `HeHcfDl.cc` | 396 | DL MU scheduling, per-STA queues, fallback decisions. |
| `HeHcfUl.cc` | 660 | UL trigger construction, BSR/BSRP handling, triggered response processing. |
| `HeHcfTxRx.cc` | 301 | Receive/transmit hooks, implicit BSR, TX completion handling. |
| `HeHcf.h` | 145 | Shared state and public/protected integration surface. |

This is a real improvement. The class is still large as a conceptual owner, but the source files now follow responsibility boundaries and are easier to review.

Remaining concern: many responsibilities still meet in a single class API. Further decomposition could extract UL response state, BSR accounting, and DL candidate staging into owned helper classes.

### 3.2 Frame Sequences

HE-specific frame-sequence code is substantial:

- `HeDlMuTxOpFs.cc` implements DL MU PPDU construction, per-user A-MPDU packing, sequential BAR, MU-BAR, BlockAck handling, BA-window checks, TXOP/PPDU duration trimming, and MU-MIMO stream indexing.
- `HeUlMuTxOpFs.cc` implements Basic/BSRP Trigger, HE-TB response collection, RU/Trigger matching, and Multi-STA BlockAck generation.
- `HeSoundingFs.cc` handles sounding-related exchanges.
- Generic frame-sequence helpers now express sequences as composable `SequentialFs`, `OptionalFs`, `StepFs`, transmit steps, receive steps, and receive-collection steps.
- Receive timeout and unexpected-response behavior are now properties of the active receive step.

Assessment: **architecturally better than a monolithic state machine.** The grammar-style sequence composition is easier to reason about, and the HE receive-collection path is stricter and more explicit.

Remaining concern: `HeDlMuTxOpFs.cc` is 1160 lines and contains several nested helper classes plus complex packing logic. It is correct-looking and heavily commented, but it remains the highest-risk file for behavioral regressions.

### 3.3 Scheduling

The scheduling architecture is clean:

- DL scheduler interface: `IIeee80211HeDlScheduler`.
- UL scheduler interface: `IIeee80211HeUlScheduler`.
- DL implementations: equal-sized RU, backlog-based, HoL-min-delay.
- UL implementations: equal-sized RU and backlog-based.
- Common scheduler bases implement RU fitting, duration estimation, rate-control hookup, SNR estimation, and debug/watch state.
- Schedulers can be configured from NED and examples.

Assessment: **strong and reusable.** The scheduler layer is one of the best-designed parts of the implementation.

Important modeling note: the schedulers intentionally use INET heuristics for backlog-to-RU mapping, SNR estimation, duration balancing, and fairness. The standard does not specify these AP scheduling choices, so heuristic behavior is appropriate, but users should know what is normative and what is policy.

### 3.4 Management and Capability Negotiation

HE management support includes:

- HE Capabilities and HE Operation elements.
- Per-peer negotiated HE capabilities in the MIB.
- BSS color and HE operation state.
- TWT management hooks.
- 6 GHz capability element structures.
- Multi-TID aggregation, LDPC, OFDMA, MU-MIMO, channel-width, and RU-size capability intersections.

Assessment: **good.** Capability negotiation is wired into scheduler constraints and MAC feature selection, which prevents many invalid configurations from silently producing impossible transmissions.

## 6. Code Quality Assessment

### 6.1 Documentation and Standard References

Rating: **very good**.

The current HE code is heavily annotated with IEEE 802.11-2024 clause, table, and figure references. The comments also distinguish standard requirements from INET modeling choices in several important places:

- Scheduler backlog-to-RU mapping is documented as an INET heuristic.
- Scheduler SNR estimation is documented as a uniform-power approximation.
- UORA per-AC simplification is explicitly called out.
- Trigger ID is documented as a simulation correlation key rather than a literal standard field.
- HE MU container packets are documented as an internal modeling representation.

This is exactly the kind of documentation a simulation model needs. It prevents readers from confusing implementation conveniences with on-wire fields.

### 6.2 Modularity

Rating: **good and improved**.

Positive examples:

- PHY calculations are in `Ieee80211HePhyCalculator.cc/.h`, not purely header-only.
- HE HCF is split across core/DL/UL/TX-RX translation units.
- Sounding, puncturing, TWT gating, UL trigger policy, UL coordinator, CSI manager, and schedulers are separate components.
- Rate adaptation has a clean HE-specific interface.
- Frame sequences are expressed through composable sequence/step primitives.

Remaining concerns:

- `HeDlMuTxOpFs.cc` is still very large and carries high coupling.
- `HeHcf` remains the owner for many orthogonal states.
- `Ieee80211HeSigCodec.h`, `Ieee80211HeRu.h`, and large parts of `Ieee80211HeMode.h` are still heavy headers.
- Some helpers are anonymous namespace functions/classes inside large `.cc` files, limiting direct unit testing.

### 6.3 Defensive Programming

Rating: **good**.

The implementation uses:

- `ASSERT()` heavily around scheduler invariants, RU uniqueness, AID/RU matching, duration constraints, and state assumptions.
- Non-throwing PHY validation results for complex PPDU calculations.
- Runtime validation of rate-control and scheduler NED parameters.
- Strict receive filtering for HE TB responses.
- Graceful fallback from HE MU to single-user HCF paths when not enough eligible users remain.

One risk remains: some code paths are defensive only through runtime assertions. In release-like simulation use, invalid dynamic states can still surface as dropped opportunities or warnings rather than structured test failures.

### 6.4 Observability

Rating: **excellent**.

The code now includes extensive WATCH instrumentation across HE MAC and PHY state:

- HE HCF summary, pending UL trigger, triggered UL exchanges, station queue banks, CSI summary.
- Scheduler last candidates, allocations, rejected candidates, punctured subchannels, common duration, scheduling reason.
- UL coordinator BSR map, fresh/backlogged report counts, OBO/OCW, trigger counters, random-access events.
- Rate-control selected MCS/NSS, probing, EWMA probability.
- Receiver PHY state and other HE-specific debug state.

This makes Qtenv inspection much better. It also makes example walkthroughs more credible because users can inspect live state instead of reverse-engineering packet logs.

## 7. Codebase Integration Analysis

This section focuses on how the pre-existing C++ codebase was updated to host the HE implementation. The distinction is important: much of 802.11ax is implemented in new HE-specific modules, but the implementation also changes shared MAC, management, PHY, radio-medium, serializer, and rate-selection paths that are used by older 802.11 modes.

The comparison point for this inventory is the same pre-HE baseline used above, `1434013bd6f33f8287555866d7f6585e66193540`.

### 7.4 Compatibility Assessment

The implementation is mostly compatibility-conscious:

- HE-specific behavior is generally gated by `modeSet == "ax"`, `HeHcf`, HE PHY headers, or HE request/receive tags.
- Base `Hcf` returns the shared EDCAF queue for `getPerStaQueue()`, so per-STA queueing is inactive unless `HeHcf` overrides it.
- TWT is inactive unless a TWT module is configured and enabled.
- Parallel reception is gated by `supportsParallelReception()` and currently returns true only for HE trigger-based uplink PPDUs.
- LDPC mode equivalence in `Ieee80211ModeSet` is intended to let receivers accept LDPC variants without requiring every LDPC object in the mode set.

The main compatibility risks are therefore concentrated in shared fixes/generalizations rather than HE branches:

- BlockAck and sequence-control serialization changes can affect older BlockAck scenarios.
- A-MPDU delimiter parsing and deaggregation can affect HT/VHT aggregate reception.
- Bandwidth-aware mandatory-rate selection can affect old mode sets when radio/transmitter bandwidth is present.
- Management frame length/element changes can affect association/probe/beacon behavior even before HE data transfer begins.
- Common scalar partial-overlap behavior changes interference semantics globally.

### 7.5 Overall Integration Verdict

The update is architecturally reasonable: new HE behavior is mostly implemented in new HE classes, and the original codebase was generalized where the old abstractions were too narrow for multi-user PHY/MAC behavior. The integration is not a superficial bolt-on; it extends core contracts in the places where 802.11ax genuinely needs new semantics: frame sequences, BlockAck, management capabilities, PHY headers, radio reception, schedulers, and mode selection.

The cost is broader regression exposure. The most important next testing work is not another HE-only unit test; it is a set of legacy regression tests around the original paths that were generalized for HE: BlockAck serialization, A-MPDU receive/deaggregation, HCF timeout handling, management association frames, and mode/rate selection for a/g/n/ac.

| ID | Threat | Risk |
|---|---|---|
| T1 | Complex interaction between DL MU packing, BlockAck windows, TXOP limits, and rate selection. | Behavioral regressions may be subtle and scenario-dependent. |

## 11. Final Assessment

The current 802.11ax implementation is in good shape. It now has the features a user would expect from a serious packet-level Wi-Fi 6 model: DL/UL OFDMA, HE MU/TB PHY headers, standard RU encoding, BlockAck machinery, BSS coloring, TWT, preamble puncturing, LDPC/DCM/PE, UORA, BSR, ER SU, MU-MIMO, and adaptive HE rate control.

The code is also much better documented and more inspectable than a typical feature branch of this size. The standard annotations are unusually helpful, and the WATCH/statistic instrumentation makes the model practical to debug in Qtenv.

The implementation is not "done" in the sense of research-grade validation. It still needs fingerprint tests, stronger integration tests for new adaptive behavior, and external validation for PHY/error/rate-control assumptions. But as an INET packet-level feature implementation, it is broad, coherent, and increasingly maintainable.

**Bottom line:** the project has moved from "major HE implementation with notable gaps" to "substantial, usable 802.11ax model with remaining validation and maintainability work." The next best investment is regression protection, especially fingerprints and targeted tests around HE Minstrel, BSR, and DL MU packing.

> [!NOTE]

>
> 1. **HE Rate Adaptation, ER SU, and BSR Accounting** (`d339bf67f4`) — added `HeMinstrelRateControl` (Minstrel-style per-peer MCS/NSS), `HE_EXTENDED_RANGE_SU` PPDU format with duplicated HE-SIG-A, and centralized BSR HT Control packing/parsing with AP-side backlog accounting signals.

>
> Additionally, five documentation/annotation passes (`1071c52`, `cc10788`, `0f62aef`, `bb88815`, `7f01493`) added standard-referencing comments across PHY and MAC, and extensive WATCH instrumentation was added across the HE path.
>
> The previous review's struck-through weaknesses (W5, W7, W9, W11, W12) remain fixed. Weaknesses W1, W2, W3 have been **fully or substantially resolved**. Three new weaknesses identified.

### 1.6 PHY Layer — Error Model

The NIST and Yans error models have been extended with HE-specific branches:

- **Per-MCS modulation mapping** correctly maps MCS 0–11 to BPSK/QPSK/QAM-16/64/256/1024 in [getHeDataSuccessRate()](src/inet/physicallayer/wireless/ieee80211/packetlevel/errormodel/Ieee80211NistErrorModel.cc).
- **LDPC SNR boost**: A flat **1.5 dB** multiplicative gain (`std::pow(10.0, 1.5 / 10.0)`) is applied to LDPC-coded data.

> [!WARNING]
> The 1.5 dB LDPC coding gain is a **reasonable simulation approximation** used in ns-3, but is not directly mandated by the standard. It varies by MCS and code rate in practice. This should be documented as a modeling choice. The code still lacks an inline comment flagging this as a modeling assumption (W10 remains open).

### 2.2 Strengths

- **DL OFDMA** provides 7 configurations covering different scheduler strategies, bandwidth scaling (20→80 MHz), MU-MIMO, and Multi-TID — this is the most comprehensive example.
- **HE Features** is well-designed as a feature matrix: it isolates individual HE PHY features (LDPC, PE, puncturing) and then combines them.
- **BSS Coloring** provides a clear A/B comparison (disabled vs. enabled) with two overlapping BSSes.
- **TWT** expanded from 1 to 4 configurations (Baseline, IndividualUnannounced, IndividualAnnounced, Broadcast).
- **HE Rate Adaptation** provides a clean A/B comparison between fixed MCS and Minstrel-style adaptive MCS, with 4 STAs at varying distances from the AP. **(NEW)**
- **HE ER SU** demonstrates the range improvement of ER SU at the cell edge with a baseline SU comparison. **(NEW)**
- **HE BSR** is the most instructive new example, with 3 configs contrasting full BSR accounting, stale-BSR behavior, and implicit BSR via SU QoS Data — demonstrating that implicit BSR achieves the highest throughput (1419 packets) with zero BSRP triggers. **(NEW)**
- All new examples have `description` on every config and follow workspace `.ini` rules.
- Most examples now include `walkthrough.md` documentation. **(NEW)**

### 2.3 Weaknesses

- **Short simulation times** (0.5–2 s). These are suitable for functional testing but insufficient for meaningful throughput/latency statistics.
- **No parameter studies.** No `repeat`, no `${variableName=...}` iterators. Cannot demonstrate capacity gains without running a campaign manually.
- **Missing `description` on 3 UL OFDMA configs**: `ScheduledOnly`, `MixedUora`, `EqualRus` in `ul_ofdma/omnetpp.ini` still lack descriptions.
- **No visualization configuration** beyond HE MU signal labels. No throughput visualizer, no data link visualizer, no statistics collection configuration for `opp_scavetool` post-processing.
- **No fingerprint tests** attached to any example. The examples are not locked against regressions.

## 3. Code Quality

### 3.1 Comments & Documentation

**Rating: ★★★★★ (Excellent — upgraded since v2)**

- Standard references are pervasive: nearly every struct, function, and constant carries an IEEE 802.11-2024 clause/table citation. Examples: every field in `Ieee80211HePhyCalculator.h`, every RU helper in `Ieee80211HeRu.h`.
- Class-level doxygen comments explain purpose, algorithms, and constraints (e.g., [HeHcf](src/inet/linklayer/ieee80211/mac/coordinationfunction/HeHcf.h#L24-L38), [computeHePpduParameters](src/inet/physicallayer/wireless/ieee80211/packetlevel/Ieee80211HePhyCalculator.h#L252-L259)).
- Inline comments explain *why*, not just *what*.
- **Five standard-annotation passes** (`1071c52`, `cc10788`, `0f62aef`, `bb88815`, `7f01493`) added corrected standard references across the entire PHY and MAC codebase, explicitly flagging non-compliant approximations. **(NEW)**
- The UORA implementation in `HeUlCoordinator` is well-documented with standard clause references (26.5.4, 26.5.4.3).
- BSR implementation comments reference Clause 9.3.1.22 and 26.5.2.2. **(NEW)**

**Gaps:**
- The MCS table static declarations in `Ieee80211HeMode.h` (735 lines) have no per-entry comments and could benefit from code generation or a table-driven approach.
- The 1.5 dB LDPC coding gain in the error model still lacks an inline comment marking it as a modeling assumption (see W10).

### 3.2 Programming Practices

**Rating: ★★★★★ (Excellent — upgraded since v2)**

**Positive:**
- Non-throwing validation API: `Ieee80211HePhyValidationResult` with `operator bool()` and descriptive `.error` strings is excellent.
- Liberal use of `ASSERT()` in the MAC layer.
- `EV_INFO` / `EV_DEBUG` logging at decision points throughout HeHcf.
- `DelayedInitializer` for the MCS table avoids static initialization order issues.
- Clean separation between scheduling interface (`IIeee80211HeDlScheduler`) and implementations.
- **`IReceiveStep` timeout/validation policies** — receive timeout behavior (`ABORT_SEQUENCE` vs `COMPLETE_STEP`), timeout completion state, response validation, and unexpected-response handling are now step-level policies. This eliminated the `dynamic_cast<HeDlMuTxOpFs *>` special case from `HeFrameSequenceHandler`. **(NEW)**
- **Grammar-style frame sequences** — `StepFs` and `IndexedRepeatingFs` generic helpers express HE DL/UL MU exchanges declaratively, closer to the IEEE 802.11 frame-sequence grammar. **(NEW)**
- **Extensive WATCH instrumentation** — HeHcf, HeUlCoordinator, and Ieee80211Receiver expose key internal state (trigger type, OBO counter, BSR summary, CSI table, exchange count) as WATCH variables visible in Qtenv. **(NEW)**
- **HeHcf decomposed into 4 compilation units** — core (350 LOC), DL (396 LOC), UL (660 LOC), Tx/Rx (301 LOC). No single file exceeds 660 lines. **(NEW)**
- **Rate control contract** — `IIeee80211HeRateControl` interface cleanly separates MCS/NSS selection (`selectHeMode`) from feedback (`reportHeTxResult`, `reportHeRxSnir`). **(NEW)**
- **BSR helpers** — centralized `packHeBufferStatusHtControl()` / `unpackHeBufferStatusHtControl()` in [Ieee80211HeBsr.h](src/inet/linklayer/ieee80211/mac/Ieee80211HeBsr.h) replace scattered bit manipulation. **(NEW)**

**Remaining concerns:**
- **`Ieee80211HeRu.h` (372 lines) and `Ieee80211HeSigCodec.h` (629 lines) remain fully header-based.** Together these two headers total ~1 000 lines of `inline` code.
- **MCS table is manually enumerated** — 384+ static members typed out individually. This is maintenance-heavy and error-prone.

### 3.3 Clarity & Readability

**Rating: ★★★★★ (Excellent — upgraded since v2)**

- Naming conventions are consistent with INET (PascalCase classes, camelCase methods/variables).
- Enum values use descriptive names (`HE_GI_0_8_US`, `HE_CODING_LDPC`, `HE_EXTENDED_RANGE_SU`).
- The scheduler interface is clean and self-documenting via `CandidateInfo` and `ScheduleContext` structs.
- Decomposed modules (`HeSoundingCoordinator`, `HePreamblePuncturing`, `HeTwtGating`) have clear single-purpose interfaces.
- **HeHcf split files** are semantically named: `HeHcfDl.cc` (DL scheduling), `HeHcfUl.cc` (UL triggering), `HeHcfTxRx.cc` (callbacks). **(NEW)**
- **`ReceiveStep` constructor** is explicit about all policy dimensions: timeout handling, validator, unexpected-response handling, timeout completion. **(NEW)**

### 3.4 Reusability

**Rating: ★★★★★ (Excellent — upgraded since v2)**

- The scheduler is **fully pluggable**: `IIeee80211HeDlScheduler` and `IIeee80211HeUlScheduler` interfaces with NED-configurable implementations.
- Three DL schedulers (EqualSizedRUs, BacklogBased, HoLMinDelay) and two UL schedulers (EqualSizedRUs, BacklogBased) are provided.
- The PHY calculator is a pure function — no module dependencies.
- `Ieee80211HeCapabilities` struct + `negotiateHeCapabilities()` are reusable without MAC module coupling.
- **HeSoundingCoordinator** is a standalone `cSimpleModule` submodule, independently testable and configurable via NED.
- **HePreamblePuncturing** is a self-contained validation utility with no module dependencies.
- **`IIeee80211HeRateControl`** is a reusable interface shared by SU rate selection and HE MU schedulers. Implementations can be NED-configured independently. **(NEW)**
- **`StepFs` / `IndexedRepeatingFs`** are generic frame sequence building blocks usable by any frame sequence, not just HE. **(NEW)**

### 3.5 Debugging Ease

**Rating: ★★★★★ (Excellent)**

- Rich `EV_INFO` / `EV_DEBUG` logging throughout the MAC scheduling path.
- `ASSERT()` guards at critical decision points.
- Custom Qtenv visualization for HE MU signals (`displayHeMuSignalDetails`, `displayHeMuSignalPhyFields`).
- Non-throwing validation result with diagnostic error strings in PHY calculations.
- **UORA logging** with backoff state, OCW, and RA-RU selection details.
- **Statistical signals** for UORA events (`heUlRandomAccessAttempt`, `heUlRandomAccessSuccess`).
- **BSR statistical signals** (`heUlBufferStatusReportedBytes`, `heUlBufferStatusScheduledBytes`, `heUlStaleBufferStatus`, `heUlBsrpTriggerSent`, `heUlBasicTriggerSent`). **(NEW)**
- **Rate control signals** (`heRateSelectedMcs`, `heRateSelectedNss`, `heRateProbe`, `heRateSuccessProbability`). **(NEW)**
- **WATCH instrumentation** across HeHcf (trigger type, exchange count, queue bank count, CSI summary, HCF summary), HeUlCoordinator (OBO counter, contention window, buffer status summary, fresh/backlogged report counts), and Ieee80211Receiver (last HE reception details, spatial reuse status). **(NEW)**

## 6. Architecture Improvements Since v2

### 6.1 HeHcf Full Decomposition **(NEW since v2)**

The HeHcf class has been **fully decomposed** from a single 1265-line `.cc` file into 4 semantically-named compilation units:

| File | Lines | Responsibility |
|---|---|---|
| [HeHcf.cc](src/inet/linklayer/ieee80211/mac/coordinationfunction/HeHcf.cc) | 350 | Core lifecycle, initialization, timer handling, WATCH setup |
| [HeHcfDl.cc](src/inet/linklayer/ieee80211/mac/coordinationfunction/HeHcfDl.cc) | 396 | DL MU scheduling, start/fallback, schedule context collection |
| [HeHcfUl.cc](src/inet/linklayer/ieee80211/mac/coordinationfunction/HeHcfUl.cc) | 660 | UL trigger scheduling, trigger response handling, BSR parsing, Multi-STA BA cleanup |
| [HeHcfTxRx.cc](src/inet/linklayer/ieee80211/mac/coordinationfunction/HeHcfTxRx.cc) | 301 | Transmit/receive callback overrides, implicit BSR injection, legacy preamble handling |

**Net result:** No single file exceeds 660 lines. The receive path is split into named helpers: trigger processing, triggered BlockAck response, pending triggered UL retry, triggered UL response packet construction, Multi-STA BlockAck processing, and preamble-puncturing capability check.

The previous extraction of `HeSoundingCoordinator` (324 lines), `HePreamblePuncturing` (131 lines), and `HeTwtGating` (24 lines) into standalone submodules is preserved.

### 6.2 Frame Sequence Grammar Refactoring **(NEW since v2)**

Frame sequences were refactored to express HE MU exchanges closer to the IEEE 802.11 frame-sequence grammar:

| Component | Change |
|---|---|
| `StepFs` | New generic single-step frame sequence: prepare + complete lambdas. Replaces ad-hoc step logic. |
| `IndexedRepeatingFs` | New factory-based indexed repeating frame sequence. Replaces manual user-iteration loops. |
| `HeDlMuTxOpFs` | Rewritten as grammar-style composition: HE-MU-PPDU → (MU-BAR + HE-TB BlockAck ∣ repeated BlockAckReq + BlockAck) |
| `HeUlMuTxOpFs` | Rewritten as: Trigger → HE-TB PPDU collection → Multi-STA BlockAck |

### 6.3 IReceiveStep Timeout/Validation Policies **(NEW since v2)**

Receive timeout behavior and response validation are now properties of `IReceiveStep`, eliminating type-specific special cases:

| Policy | Description | Default |
|---|---|---|
| `TimeoutHandling` | `ABORT_SEQUENCE` or `COMPLETE_STEP` | `ABORT_SEQUENCE` |
| `Completion getTimeoutCompletion()` | Result state on timeout | `EXPIRED` |
| `isExpectedResponse()` | Response validator lambda | Always true |
| `UnexpectedResponseHandling` | `IGNORE_RESPONSE` or `REJECT_STEP` | `REJECT_STEP` |

The HE-specific `dynamic_cast<HeDlMuTxOpFs *>` timeout special case has been removed from `HeFrameSequenceHandler.cc` (reduced to 18 lines). Response validators for CTS/ACK/BlockAck were added to `PrimitiveFrameSequences.cc`.

## 9. Summary

The 802.11ax implementation is a **substantial, correct, and mature** addition to INET, covering all key HE features (OFDMA, MU-MIMO, LDPC, BSS Coloring, TWT, preamble puncturing, UORA, **rate adaptation, ER SU, and full BSR accounting**) with strong standard grounding and a clean, decomposed architecture. The PHY numerology is correct. The test suite (39 unit tests + 10 shared-scope, 6 900+ LOC) is comprehensive.

Since the v2 review, the implementation has reached a higher level of maturity:

1. ✅ **HE rate adaptation fully implemented** — `HeMinstrelRateControl` with per-peer EWMA, probing, SNR seeding, ER SU/DCM constraints, and scheduler integration
2. ✅ **HE Extended Range SU modeled** — correct preamble timing with duplicated HE-SIG-A, PHY serialization, and rate-control integration
3. ✅ **BSR fully implemented** — explicit (BSRP) + implicit (SU QoS Data) with AP-side accounting, freshness tracking, and 6 statistical signals
4. ✅ **HeHcf fully decomposed** — split into 4 semantically-named compilation units; no single file exceeds 660 lines
5. ✅ **Frame sequences refactored** — grammar-style `StepFs`/`IndexedRepeatingFs` helpers; `IReceiveStep` timeout/validation policies eliminate `dynamic_cast` special cases
6. ✅ **Extensive WATCH instrumentation** — 20+ WATCH expressions for Qtenv debugging across HeHcf, HeUlCoordinator, and Receiver
7. ✅ **5 standard-annotation passes** — corrected IEEE 802.11-2024 references across the entire PHY and MAC codebase
8. ✅ **3 new example simulations** — rate adaptation, ER SU, BSR (7 new config variants)

**Remaining open items (in priority order):**
1. **Add fingerprint tests** to lock example simulation outputs against regressions.
2. **Add parameter studies / showcases** for quantitative HE gains demonstration.
3. **Move remaining header-only code** (`Ieee80211HeRu.h`, `Ieee80211HeSigCodec.h`) to .cc files.
4. **Document modeling assumptions** (LDPC 1.5 dB gain) with inline comments.
5. **Add descriptions to 3 UL OFDMA example configs** (`ScheduledOnly`, `MixedUora`, `EqualRus`).

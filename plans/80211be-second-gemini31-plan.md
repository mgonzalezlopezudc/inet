# 802.11be (EHT) and Multi-Link Operation (MLO) Implementation Plan

This plan details the implementation of the 802.11be (EHT) MAC layer and Multi-Link Operation (MLO) within the OMNeT++ INET framework. It addresses specific questions regarding queue management, scheduling reuse, and management frame alignment.

## 1. MLO MAC Architecture: Upper and Lower MAC

The 802.11be standard splits the MAC into an Upper MAC (shared across links) and Lower MACs (one per link).

### Queueing (Global vs Per-Link)

**Standard Behavior**: In 802.11be, the queues are maintained per-AC (VO, VI, BE, BK) at the **MLD level**. Each affiliated STA (Lower MAC) pulls from these MLD-level queues according to the TID-to-Link Mapping.

**Implementation**: 
- `Ieee80211MldMac` will instantiate the 4 standard QoS queues (`pendingQueue[4]`).
- Each Lower MAC (`Ieee80211Mac`) will configure its `Edca` -> `Edcaf` submodules to use a reference (via NED module path) to the `Ieee80211MldMac` queues instead of instantiating their own.
- OMNeT++ `PacketQueue` handles concurrent pop/push safely.

### NED Modules and C++ Classes

#### [MODIFY] `inet/linklayer/ieee80211/mac/Ieee80211MldMac.ned` and `.h/.cc`
- **Role**: Upper MAC. Holds the `mldMacAddress` and the `pendingQueue` instances.
- **Responsibilities**:
  - Sequence Number Assignment (Global SN space for the MLD).
  - Global BlockAck buffering.
  - Exposing the `pendingQueue` array for Lower MACs.
- **Traffic Steering**: The steering happens implicitly if multiple links point to the same global `pendingQueue`. When a link's HCF wins contention, it pops from the global queue. If a TID is restricted to specific links, we will implement an `IMloTrafficSteeringPolicy` filter on the popping operation.

#### [MODIFY] `inet/linklayer/ieee80211/mac/Ieee80211Mac.ned`
- **Role**: Lower MAC.
- **Changes**: Receives an `mldMacModule` string parameter. If set, it bypasses local sequence numbering and queue instantiation, deferring to the MLD MAC.

## 2. EHT Schedulers and Frame Sequences

> [!NOTE]
> The HE MU schedulers (`HeDlMuTxOpFs`, `HeUlMuTxOpFs`) will be reused and extended for EHT.

### Subclassing vs Configuration
Since EHT MU uses the same underlying principles as HE MU (trigger frames, BSR, OFDMA) but with different PHY headers (EHT-SIG) and MRU configurations:
- We will create **`EhtDlMuTxOpFs`** inheriting directly from **`HeDlMuTxOpFs`**.
- We will create **`EhtUlMuTxOpFs`** inheriting directly from **`HeUlMuTxOpFs`**.
- The overridden methods will be `createMuRts()`, `createTriggerFrame()`, and the PHY header generation logic to ensure EHT capabilities and MRU allocations are used instead of HE ones.

## 3. Management Frames and Serializers

> [!IMPORTANT]
> As requested, EHT Management Elements will be structured identically to existing HE/VHT elements, ensuring codebase consistency.

### Structure

#### [NEW] `inet/linklayer/ieee80211/mib/Ieee80211EhtCapabilities.h`
- A C++ struct representing the MIB/internal state of EHT capabilities (matching `Ieee80211HeCapabilities.h`).
- Contains `rxMcsNss`, `txMcsNss`, EHT PHY flags, and MAC flags.

#### [NEW] `inet/linklayer/ieee80211/mgmt/Ieee80211EhtMgmtElements.h`
- Provides inline conversion functions: `makeEhtCapabilitiesElement(const Ieee80211EhtCapabilities& cap)` and `makeEhtCapabilities(const Ieee80211EhtCapabilitiesElement& el)`.
- Mirrors `Ieee80211HeMgmtElements.h`.

#### [MODIFY] `inet/linklayer/ieee80211/mgmt/Ieee80211MgmtFrameSerializer.cc`
- Serializes/deserializes the `Ieee80211EhtCapabilitiesElement` and `Ieee80211MldElement` directly into the byte stream.
- Maintains strict 802.11 wire compliance.

## 4. Verification Plan

### Automated Tests
- Run INET's internal 802.11 unit tests to ensure no HE/AC functionality is broken by the MLD MAC split.
- Create a new `EhtMloTest` scenario in `tests/fingerprint/ieee80211/` running an AP MLD and STA MLD on two 5GHz and 6GHz channels, verifying throughput aggregation.

### Manual Verification
- Review PCAP files generated by OMNeT++ to verify that EHT Action frames, Association Requests, and EHT-SIG PHY headers are correctly formatted at the byte level.
